# JobTracker Redesign: Backend Capability Inventory

This inventory compares the current redesign product surface with registered backend routes, service composition, requirements, and focused tests in the shared worktree.
Endpoint registration alone is not treated as proof that a user workflow is complete.
The legacy `/dashboard`, `/setup`, `/features`, and `/legacy/*` routes are cited when they expose a capability that the redesign does not.

## Verified current baseline

The redesign uses `GET|PUT /config/providers`, and a successful provider change calls `POST /config/providers/llm/health`.
The health route resolves configured Azure OpenAI and Ollama adapters through normal dependency injection rather than an unconditional unavailable stub.
Evidence: `backend/tests/test_provider_config_api.py::test_provider_config_update_reconfigures_active_scheduler_before_settings`, `backend/tests/test_llm_provider_health.py::test_health_endpoint_resolves_real_azure_adapter_through_normal_di`, and `frontend/src/App.test.tsx::redesign settings confirms a provider only after success and checks unavailable health`.

The redesign delete-everything action uses `POST /local-data/wipe` with the exact `wipe-local-data` confirmation phrase.
Evidence: `backend/tests/test_wipe_data_api.py::test_wipe_data_endpoint_deletes_configured_local_data` and the `wipeDataLocalDataWipePost` call in `frontend/src/redesign/pages/SettingsPage.tsx`.

The redesign overview renders the authoritative five-stage response from `GET /metrics/funnel` with separate loading, empty, and error states.
Evidence: `backend/tests/test_metrics_funnel_api.py::test_metrics_funnel_returns_approved_stage_contract`, `frontend/src/App.test.tsx::renders the exact five authoritative funnel stages from metrics funnel`, and `frontend/src/App.test.tsx::distinguishes funnel loading, successful empty, and typed failure`.

The redesign detail page exposes audited status and event correction through `PATCH /applications/{application_id}/status` and `PATCH /applications/{application_id}/events/{event_id}`.
Status evidence: `backend/tests/test_manual_application_edit_api.py::test_patch_application_status_edits_status_with_audit` and `frontend/src/redesign/pages/DetailPage.test.tsx::keeps the saved status until the audited correction succeeds and disables duplicates`.
Event evidence: `backend/tests/test_manual_application_edit_api.py::test_patch_application_event_edits_timeline_event_with_audit` and `frontend/src/redesign/pages/DetailPage.test.tsx::edits an event in its card through the audited endpoint`.

The app factory composes a configured sync job by default, the scheduler can enable that job at runtime, and `PUT /config/providers` applies scheduler changes before mutating settings.
Evidence: `backend/tests/test_app_factory.py::test_create_app_uses_configured_sync_job_by_default`, `backend/tests/test_sync_service.py::test_sync_scheduler_reconfigure_enables_live_interval_job`, and `backend/tests/test_provider_config_api.py::test_provider_config_update_reconfigures_active_scheduler_before_settings`.

The redesign developer screen is a static projection of `frontend/src/featureStatus/featureStatusRegistry.ts`; it is not a runtime backend capability probe.
Evidence: `frontend/src/App.test.tsx::redesign developer derives truthful statuses from the feature registry` verifies registry lookup and fallback behavior only.
