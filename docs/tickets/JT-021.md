# JT-021 - Manual Override Schema Migration

JT-021 adds the SQLite schema support for audited manual aggregation overrides.

## What Changed

The Alembic revision `20260705_0005_application_corrections.py` creates the `application_corrections` table.
The table records one audit row per manual correction against an application.

Columns:

- `id` - integer primary key.
- `application_id` - required foreign key to `applications.id` with cascade delete.
- `correction_type` - required enum-like text constrained to `merge`, `split`, `status_edit`, `event_edit`, or `reset_lock`.
- `before_json` - required JSON text snapshot of the corrected state before the override.
- `after_json` - required JSON text snapshot of the corrected state after the override.
- `reason` - optional human-readable reason for the correction.
- `created_at` - required correction timestamp.

The migration also adds `ix_application_corrections_application_created_at` so correction history can be read efficiently for one application in timestamp order.

## Why

FR-2.8 requires manual correction support for aggregation mistakes.
The product invariant is that factual answers come from the clean `applications` table and its `application_events` timeline.
Manual edits must therefore be auditable, and automatic aggregation must later be able to respect user corrections without silently overwriting them.
