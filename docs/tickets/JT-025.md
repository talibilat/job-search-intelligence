# JT-025 - Pydantic Domain DTOs

## What Changed

JT-025 adds focused Pydantic DTO modules for the core JobTracker domains that cross repository, pipeline, service, and API boundaries.
The package now exposes typed DTOs for raw emails, email classifications, applications, application events, insights, email chunks, application corrections, and chat messages.
The existing `app.models.records` import path remains an aggregate module so existing repository code can keep importing stable names while newer code can import from focused modules.

## Why

The Phase 0 architecture requires Pydantic v2 DTOs at every boundary.
These DTOs preserve the local-first SQLite model and the invariant that factual job-search answers come from `applications` plus `application_events`.
The split modules reduce merge collisions for future tickets that need to import or extend one domain shape without editing a single large shared file.

## Notable Invariants

`RawEmailRecord` keeps retained body text out of `repr` and enforces that metadata-only emails do not carry body text.
`EmailClassificationRecord` validates the job email category enum, confidence range, model metadata, prompt version, and timezone-aware classification timestamp through the shared classification result DTO.
`ApplicationRecord` validates the non-negative salary fields and rejects inverted salary ranges.
`ApplicationEventRecord` allows a null `email_id` only for `ghost_inferred` events.
`EmailChunkRecord` keeps retained chunk content and embeddings out of `repr` and requires the 1536-dimensional sqlite-vec embedding shape from the current schema.
`ApplicationCorrectionRecord` parses JSON object columns for before and after snapshots.

## How To Verify

Run `uv run --project backend pytest backend/tests/test_domain_dtos.py -q` for the focused DTO contract tests.
Run the backend verification suite required by `AGENTS.md` before merging: Ruff, mypy, and relevant pytest tests.
