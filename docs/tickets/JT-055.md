# JT-055 - Implement Gmail Auth URL Endpoint

## What changed

JT-055 is implemented by `GET /auth/gmail` in `backend/app/api/auth.py`.
The route delegates OAuth start behavior to `backend/app/services/gmail_auth.py`, which generates and stores an OAuth state before calling the configured email provider.
The Gmail provider implementation in `backend/app/providers/email/gmail.py` reads the user-owned Desktop OAuth client JSON and builds a Google authorization URL constrained to `gmail.readonly`.
The behavior is covered by `backend/tests/test_gmail_auth_api.py`.

## Why it changed

Gmail ingestion needs a read-only OAuth start endpoint before the user can authorize their own Google account.
The endpoint must keep the app local-first, avoid shared credentials, avoid exposing the Google client secret or tokens, and return typed public-safe errors when client configuration is missing or invalid.

## Verification

- `uv run pytest tests/test_gmail_auth_api.py -q`
- `uv run pytest tests/test_ticket_docs_jt_055.py -q`

## Usage notes

- Call `GET /auth/gmail` from the local backend to receive a Google authorization URL, generated state, provider name, and requested `gmail.readonly` scope.
- The endpoint does not return the Google client secret, access tokens, refresh tokens, or authorization codes.
- The callback, token exchange, token persistence, refresh, and Gmail message access are separate Phase 1 behaviors implemented by later tickets.
