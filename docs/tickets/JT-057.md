# JT-057 - Gmail Token Refresh

## What Changed

Gmail OAuth token refresh is implemented in `GmailEmailProvider`.
The provider reads the existing OAuth payload from the configured `SecretStore`, exchanges the stored refresh token for a new access token, validates that the granted scope remains `gmail.readonly`, and writes the refreshed token payload back through `SecretStore`.
The returned `EmailConnection` contains only non-secret metadata such as the updated credential expiry timestamp.

## Why

Phase 1 Gmail ingestion needs long-running backfill and incremental sync to keep working after Google's short-lived access tokens expire.
Refreshing the access token through the provider keeps token material behind the existing encrypted-at-rest secret boundary and avoids exposing secrets through API DTOs, logs, SQLite connection metadata, or sync state.

## Behavior

When Gmail metadata listing or retained-body fetching runs with an expired stored token payload, the provider refreshes the token before calling Gmail.
The refresh flow preserves the existing refresh token because Google refresh responses normally return a new access token, not a new refresh token.
If the stored credential is missing a refresh token, the provider raises a public-safe reconnect error instead of exposing token payload details.
The Gmail scope remains constrained to `https://www.googleapis.com/auth/gmail.readonly`.
