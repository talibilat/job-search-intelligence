# JT-061 - Broad Candidate Query Strategy

## What Changed

JT-061 added a provider-neutral broad job-search candidate query strategy for Phase 1 ingestion.
The backend now exposes `EmailCandidateQuery` with the `broad_job_search` strategy, static ATS and recruiter sender-domain signals, subject keyword signals, excluded label terms, deterministic score evaluation, and same-page thread promotion without exposing raw provider thread IDs.

The query is evaluated over normalized metadata after provider listing instead of becoming a Gmail-specific search filter.
That keeps broad metadata backfill intact while allowing downstream retained-body fetching to select likely job-search messages.

## Why It Changed

The product needs to retain body text for likely job-search messages without relying on brittle narrow Gmail search syntax.
This supports `FR-1`, `FR-6`, `NFR-1`, `NFR-4`, `NFR-7`, and the Phase 1 ingestion requirement to keep metadata ingestion broad while limiting private body retention to selected candidate, reconciliation, or debugging messages.

## Usage Notes

Use `build_broad_candidate_query()` from `app.pipeline.filter` when sync or retained-body selection needs the default broad job-search heuristic.
Callers should persist or inspect the public-safe decision reason and signal tokens, not raw subjects, bodies, snippets, thread IDs, or provider payloads.
Provider metadata listing requests must remain metadata-only and must not accept candidate filters.
