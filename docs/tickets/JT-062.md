# JT-062 - Retained Body Fetching

## What Changed

JT-062 added retained Gmail body fetching behind the provider seam for caller-selected message references.
The Gmail provider fetches full message payloads for selected candidate, reconciliation, or debugging messages, extracts retained body text, normalizes plain text and HTML body content, ignores attachments, and returns typed per-message body results.

Manual sync now uses the broad candidate decision path to request retained bodies for selected job-search candidates and same-page candidate thread siblings after metadata persistence.
Fetched retained bodies are persisted through the raw email repository with a retained body state instead of being mixed into metadata-only listing DTOs.

## Why It Changed

Phase 1 ingestion must store metadata for all Gmail messages while retaining body text only when a message is a likely job-search candidate or is explicitly needed for reconciliation or debugging.
This supports `FR-1`, `FR-6`, `NFR-1`, `NFR-4`, `NFR-7`, and the privacy rule that broad metadata backfill stays separate from retained body fetching.

## Usage Notes

Use `EmailProvider.fetch_message_bodies()` only with caller-selected `EmailMessageRef` values.
Do not add body text, snippets, or body-derived filters to metadata listing requests.
Retained body fetch results should be persisted only after metadata writes for the page succeed, and retained body failures should remain public-safe without exposing provider payloads or private email content.
