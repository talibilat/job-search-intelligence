# JT-065 - Raw Email Retention Flags

## What Changed

JT-065 added explicit raw email body retention state tracking at the backend DTO and synthetic fixture boundaries.
Raw email records now use the shared `RawEmailBodyRetentionState` enum with `metadata_only`, `retained`, and `debugging` values.

DTO validation rejects metadata-only rows that include `body_text` and rejects retained or debugging rows that omit `body_text`.
Downstream code can use `RawEmailRecord.has_retained_body` instead of inspecting enum values directly, and raw email repr output redacts retained body text.

Synthetic fixtures now reuse the shared retention enum and reject obsolete or inconsistent retention values.

## Why It Changed

Phase 1 ingestion needs to distinguish metadata-only records from retained job-search candidate bodies and explicit debugging or reconciliation bodies.
This supports `FR-1`, `FR-6`, `NFR-1`, `NFR-4`, `NFR-7`, and the privacy rule that email bodies are retained only when selected and never exposed accidentally through DTO repr output.

## Usage Notes

Use `RawEmailBodyRetentionState.METADATA_ONLY` when no body text is retained.
Use `RawEmailBodyRetentionState.RETAINED` for selected job-search candidate bodies and `RawEmailBodyRetentionState.DEBUGGING` for explicitly retained debugging or reconciliation bodies.
Use `RawEmailRecord.has_retained_body` when downstream pipeline code needs to decide whether body text is available.
