# JT-067 - Full Backfill Service

## What Changed

JT-067 adds a full-backfill orchestration path for Gmail ingestion.
The service loops provider metadata pages, writes metadata-only `raw_emails`, fetches retained bodies only for broad job-search candidates, and records durable backfill page progress after the page writes complete.

## Why

Phase 1 requires historical Gmail backfill to be resumable for large mailboxes while keeping metadata listing separate from retained body fetching.
JT-066 added durable page state and cursor promotion.
JT-067 uses that state from the sync service and API runtime so `POST /sync` runs the full backfill path until a completed backfill promotes an incremental cursor.

## Behavior

Full backfill uses `email_backfill_state` for provider/account progress.
Each page is listed from the stored resume token.
Metadata is persisted before candidate body retention.
Page progress is recorded only after metadata and retained bodies are written.
If body retention or provider listing fails, the backfill is marked failed with a public-safe error and the page token is not advanced.
When the final page includes a replacement provider cursor, the cursor is promoted to `email_sync_state` in the same transaction as final page recording.
