# JT-076 - Sync Status Frontend Panel

## What Changed

JT-076 adds a frontend sync status panel to the overview page.
The panel reads `GET /sync/status` through the generated TypeScript API boundary and displays the current or last sync state, provider/account metadata, mode, page count, message count, raw email count, expired-cursor recovery, and last-run timestamp.
It also exposes a `Sync now` button that calls `POST /sync` and updates the panel from the returned public `EmailSyncStatus` DTO.

## Why

FR-1.5 requires manual sync and a public status endpoint that reports progress and last-run information.
The UI now makes that Phase 1 sync state visible without adding backend behavior or reading private provider payloads.
This preserves the local-first boundary from FR-6 and keeps progress counts tied to deterministic backend fields instead of inferred browser state.

## Privacy Boundary

The panel renders only the public sync status DTO fields generated from OpenAPI.
It does not expose OAuth tokens, provider cursors, page tokens, raw email content, or email snippets.
Errors are shown from the typed public API response or as generic local-backend availability messages.

## How To Verify

Run `npm run test -- src/App.test.tsx` from `frontend/` to verify the panel renders sync state, progress counts, last-run copy, expired-cursor recovery, and manual sync behavior.
Run `npm run test:smoke` from `frontend/` to verify the browser smoke path covers the sync status panel and manual sync transition.
Run `npm run check` from `frontend/` to regenerate and validate the API client, typecheck, lint, run Vitest, and build the frontend.
