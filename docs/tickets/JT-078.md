# JT-078 - Gmail Auth Frontend Flow

## What Changed

The setup page now loads `GET /setup/status` when `/setup` renders.
It enables a `Start Gmail OAuth` action when Gmail is not connected.
That action calls `GET /auth/gmail` and displays the backend-built Google authorization URL as a `Continue to Google` link.
The page also shows the requested Gmail scope so the user can confirm it remains `https://www.googleapis.com/auth/gmail.readonly` before leaving the app.
When `GET /setup/status` reports `gmail_connected: true`, the setup page shows `Gmail callback complete` and disables the OAuth start action.

The setup status backend path now checks the local `email_connections` table for the latest non-reauth Gmail connection.
It reports `gmail_connected: false` when the table has not been created yet, preserving first-launch behavior before migrations or auth have run.

## Why

JT-055 and JT-056 already added the Gmail OAuth start and callback endpoints.
JT-078 connects those endpoints to the setup UI so a user can begin read-only Gmail authorization from the app and see whether the callback stored non-secret connection metadata.

## Privacy And Security

The frontend never receives OAuth tokens, authorization codes, client secrets, or refresh tokens.
It only receives the backend-generated authorization URL, requested scopes, and non-secret setup status.
Token material remains behind the backend `SecretStore` boundary.

## How To Verify

Run `uv run pytest tests/test_setup_status.py -v` from `backend/`.
Run `npm run test -- src/App.test.tsx` from `frontend/`.
Run `npm run check` from `frontend/` for the full frontend OpenAPI, typecheck, lint, test, and build gate.
