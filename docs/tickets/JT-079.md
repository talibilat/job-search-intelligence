# JT-079 - Add Ingestion Error Handling

## What Changed

JT-079 maps Gmail ingestion failures into typed, user-actionable API errors.
The email provider boundary now exposes stable `EmailProviderErrorCode` values and `EmailProviderUserAction` hints for reconnect, insufficient scope, rate limit, temporary outage, expired cursor, invalid provider response, and generic provider failure cases.
The Gmail metadata-listing adapter converts Gmail HTTP and response-shape failures into those public-safe provider errors without exposing OAuth tokens, Gmail response bodies, or private email content.
The FastAPI error handler maps provider errors into the standard `{"error": ...}` response body with email-specific codes and a `user_action` detail.
`POST /sync` now lets provider errors reach that global handler, so clients receive typed `401`, `403`, `409`, `429`, `502`, or `503` responses instead of a generic `400`.

## Why

Phase 1 Gmail ingestion needs resumable, user-actionable failure handling.
Authorization and scope problems should tell the client to reconnect Gmail.
Expired Gmail history cursors should tell the client to restart full sync.
Rate limits and transient Gmail outages should tell the client to retry later.
Invalid provider responses and generic request failures should stay public-safe and avoid leaking provider payloads.

## How To Verify

Run the focused backend checks from `backend/`:

```bash
uv run pytest tests/test_api_errors.py tests/test_gmail_message_listing.py tests/test_sync_api.py -q
uv run ruff check .
uv run mypy .
```

Run the frontend contract check from `frontend/` if the generated OpenAPI client changes:

```bash
npm run check
```

The relevant behavior is visible through `POST /sync` when an injected or configured email provider raises an `EmailProviderError`.
The response uses the standard error shape and includes `details[0].type == "user_action"` with `field == "email_provider"`.
