# JT-082 - Implement Heuristic Keyword Signals

## What Changed

JT-082 adds an executable keyword-signal matcher to the broad email candidate query.
The matcher checks both the email subject and already-normalized retained body text for the static job-search keyword terms used by the heuristic filter.
Existing metadata matching remains unchanged: sender-domain signals and subject keywords still select broad body-retention candidates without passing body content into provider listing requests.
The query still stores only static terms and does not serialize raw email body text, snippets, or private content.

## Why

FR-2.1 requires heuristic pre-filtering before LLM classification to reduce cost while retaining likely job-search emails.
Subject keywords alone miss messages whose subject is generic but whose normalized body says things like "thank you for applying" or "we received your application".
Body-aware keyword matching gives the Phase 2 classifier a deterministic local signal over retained plain text without adding LLM calls or provider-specific search behavior.

## How To Use

Use `build_broad_candidate_query()` to construct the default static signal set.
Call `matches_metadata()` when only provider-normalized metadata is available.
Call `matches_keywords(subject=..., normalized_body_text=...)` when retained plain-text body content is available and the caller needs a keyword-only signal across subject and body.

## How To Verify

Run the focused candidate-query tests from the repository root:

```bash
uv run --project backend pytest backend/tests/test_email_candidate_query.py
```

Run the backend verification checks for Python changes:

```bash
uv run --project backend ruff check .
uv run --project backend mypy .
```

This ticket does not add an LLM prompt, model, category, or extraction schema, so there is no golden-set eval script to run in the current scaffold.
