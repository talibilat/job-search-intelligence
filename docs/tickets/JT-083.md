# JT-083 - Candidate Scoring

## What Changed

JT-083 replaces the broad filter's implicit single-message decision with an explicit scored candidate decision.
`EmailCandidateDecision` now carries a numeric `score` and public-safe `signals` tuple in addition to the stored outcome and reason.
The broad candidate query scores sender-domain, subject-keyword, excluded-label, and same-thread signals while keeping raw subjects, body text, and provider thread IDs out of serialized decisions.

## Decision Rules

Excluded labels are a hard rejection and win over every positive signal.
Sender-domain matches and subject-keyword matches each meet the default candidate threshold on their own.
When one message in a same-page metadata batch is a direct candidate, sibling messages in the same provider thread can become candidates through the static `thread_signal:candidate_thread` token.
The thread signal never stores or logs the provider-owned thread ID.

## Sync Behavior

Sync body-retention selection now evaluates each metadata page as a batch so thread siblings are retained alongside directly signaled candidate messages.
Persisted filter decisions use the same batch evaluation path, so audit rows explain thread-promoted candidates with the public-safe thread signal token.
Reconciliation and debugging refs still bypass candidate scoring through the existing explicit-ref path.
