# JT-084 - Persist Filter Decisions

JT-084 adds an audit trail for the Phase 2 heuristic email pre-filter.
Every stored decision is local-first SQLite data and is keyed by raw email ID plus filter strategy.

## What Changed

The backend now stores heuristic filter decisions in `email_filter_decisions`.
Each row stores `email_id`, `strategy`, `outcome`, `reason`, and `decided_at`.
The sync runtime writes a decision for every normalized metadata record it evaluates with the broad job-search filter.
The repository upserts by `(email_id, strategy)` so re-runs stay idempotent.

## Why

Phase 2 relies on the heuristic pre-filter to decide which messages are broad job-search candidates before downstream classification work.
Persisting the outcome and reason makes false positives and false negatives auditable without replaying provider data.
This supports FR-2, NFR-2, NFR-4, and NFR-5 by making filtering inspectable, deterministic, and re-runnable.

## Privacy Notes

Filter reasons store only static signal names, not body text or raw subjects.
Examples are `sender_domain:greenhouse.io`, `subject_keyword:interview`, `excluded_label:spam`, and `no_filter_signal`.
The filter continues to run over normalized metadata and retained bodies are still fetched only for candidate or explicit debug and reconciliation refs.
