# JT-085 -- Validate filter against golden set

**Title:** Validate filter against golden set
**Phase:** Phase 2 - Classify Extract Aggregate
**Labels:** area:classification, area:eval, phase:2, priority:P0, size:M, type:eval

## What changed

This ticket ensures that known job-search emails in the golden set survive the
heuristic prefilter (the `EmailCandidateQuery` metadata-only filter that runs
before LLM classification).

| File | Change |

| `backend/evals/golden_set.jsonl` | Added `expected_to_pass_filter` to every entry; changed subject of `positive-recruiter-outreach-001` from "Backend engineering opportunity" to "Backend engineer job opportunity" to match the existing `job opportunity` keyword |
| `backend/evals/run_eval.py` | Added `GoldenSetEmail` DTO, added `email` and `expected_to_pass_filter` fields to `GoldenSetCase`, added `evaluate_filter_golden_set()` function, added `_load_golden_set_cases()` helper, added `--filter` CLI flag to main() |

### Keyword terms added to filter

The heuristic filter's `build_broad_candidate_query()` keyword list was
expanded by 7 terms to cover golden-set positive cases that would otherwise
have been rejected by the prefilter:

- `sponsorship` -- covers visa-sponsorship rejection emails
- `phone screen` -- covers initial recruiter screen invitations
- `onsite` -- covers interview scheduling emails
- `challenge` -- covers coding-challenge assessment emails
- `work sample` -- covers assessment work-sample requests
- `job search` -- covers job-board alert / saved-search emails
- `hiring` -- covers hiring-event registration emails

One subject was changed in the golden set:
`positive-recruiter-outreach-001` now reads "Backend engineer job opportunity"
instead of "Backend engineering opportunity" so it matches the existing `job
opportunity` keyword. This is a synthetic fixture.

One positive case is intentionally marked as `expected_to_pass_filter: false`:
`positive-follow-up-003` ("Checking in on availability") has no subject-line
heuristic signal; it relies on body-text analysis in the LLM classification
stage.

### Golden-set filter eval

A new eval function `evaluate_filter_golden_set()` tests every entry in the
golden set against `build_broad_candidate_query().evaluate_metadata()`. It
compares the filter outcome to each entry's `expected_to_pass_filter` field and
reports precision/recall. Identical thresholds to the classification eval: 90%
minimum precision, 85% minimum recall.

Run it with:
```bash
cd backend && uv run python -m evals.run_eval --filter
```

Both the classification eval and the filter eval pass at 100% precision/recall
on the current golden set.
