# JT-088 - LLM Provider Health Checks

JT-088 adds the shared health-check seam for verifying configured LLM model availability before classification or other LLM-backed runs begin.
The change maps to Phase 2, FR-2, NFR-2, NFR-4, and NFR-5.

## What Changed

The `LLMProvider` protocol now includes a `health_check` method alongside `generate`.
Provider adapters receive a provider-neutral `LLMProviderHealthCheckRequest` containing the configured chat model and embedding model names.
Adapters return a typed `LLMProviderHealthCheckResponse` with one check for the configured chat model and one check for the configured embedding model.
The response rejects inconsistent status values, such as an overall `available` result that contains an unavailable model.

The backend now exposes `POST /config/providers/llm/health`.
The route validates the selected non-secret provider settings, invokes the configured provider adapter, and returns the typed health-check result.
LLM provider failures map to sanitized API error codes without exposing secrets or raw provider payloads.

The `ensure_configured_llm_provider_available` service helper is available for future classification, insight, embedding, and chat runs.
It blocks a run before any work starts when the configured provider reports unavailable models.

## Usage

Call `POST /config/providers/llm/health` after selecting an LLM provider and configuring its visible model or deployment names.
For Ollama, the request checks `ollama_chat_model` and `ollama_embedding_model`.
For Azure OpenAI, the request checks `azure_openai_chat_deployment` and `azure_openai_embedding_deployment`.

The current branch adds the shared seam and API behavior.
Concrete Azure OpenAI and Ollama adapter HTTP checks remain owned by sibling provider tickets and are not added here.
