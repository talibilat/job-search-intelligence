# JT-089 - Classification Prompt Contract

## What Changed

JT-089 adds a versioned classification prompt contract for Phase 2 classification work.
The contract builds provider-neutral `LLMGenerationRequest` values with JSON-object response format, deterministic generation options, and an explicit prompt version.
It also defines the strict Pydantic `ClassificationPromptOutput` shape that downstream classification code must validate before storing any classification or extraction result.

## Why

FR-2 requires classification and structured extraction to preserve `applications` and `application_events` as the factual source of truth.
FR-2.6 requires reproducible re-runs by storing prompt and model versions on classification rows.
FR-2.7 requires the golden-set eval to remain the gate for classification prompt, model, category, and schema changes.
NFR-5 requires typed boundaries and maintainable provider seams.

## Contract

Use `app.pipeline.classify.build_classification_prompt_request` to create the LLM request for a retained email candidate.
Use `app.pipeline.classify.parse_classification_prompt_output` on the provider response before storing or aggregating it.
Malformed JSON, unknown categories, extra fields, invalid enum values, inverted salary ranges, and contradictory non-job output are rejected by Pydantic validation.
The prompt version defaults to `v1` and is embedded in the system prompt so callers can store the same version on `email_classifications.prompt_version`.

## Verification

Run `uv run --project backend pytest backend/tests/test_classification_prompt_contract.py` for the focused contract tests.
Run backend ruff, mypy, relevant pytest, and the classification golden-set eval after changing this contract.
