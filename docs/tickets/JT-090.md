# JT-090 - Add Classification DTOs

## What Changed

JT-090 adds classification-specific Pydantic DTOs for the Phase 2 classification boundary.
`EmailClassificationCandidate` represents retained email content prepared for classification and keeps `body_text` out of repr output.
`EmailClassificationResult` represents provider-neutral classification output with reproducibility metadata.
`EmailClassificationRecord` now reuses the result validation while adding the storage `email_id`.

## Why

FR-2 requires classification into the supported job email categories with confidence, model, prompt version, and classified timestamp metadata.
The DTOs give the upcoming classification service a typed boundary before any provider output can be stored.
They also preserve the local-first data model and the `email_classifications` row contract without adding service or repository behavior in this ticket.

## Validation Rules

Classification categories must be one of the `JobEmailCategory` values.
Confidence must be in the inclusive range from `0` to `1`.
Model and prompt version strings are stripped and must not be blank.
Classification timestamps must be timezone-aware.
Candidate `sent_at` timestamps, when present, must also be timezone-aware.
Unknown fields are rejected on the new classification boundary DTOs.

## How To Verify

Run `uv run --project backend pytest backend/tests/test_classification_dtos.py -q` for the focused DTO contract tests.
Run the backend verification suite required by `AGENTS.md` before merging: Ruff, mypy, relevant pytest tests, and the classification golden-set eval when available.
