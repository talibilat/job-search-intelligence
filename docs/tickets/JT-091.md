# JT-091 - Implement LLM Classification Service

## What Changed

- Added `ClassificationService` in `backend/app/services/classification.py`.
- The service accepts retained `EmailClassificationCandidate` DTOs and calls the configured `LLMProvider` through the existing provider-neutral prompt request.
- The service validates provider output through the existing classification prompt contract before returning accepted classification and extraction DTOs.
- Malformed provider output is returned as public-safe quarantine metadata instead of a classification record.
- Token usage reported by providers is aggregated on the service result for later run-accounting integration.

## Why

JT-091 needs the core LLM classification orchestration without taking over later persistence or aggregation tickets.
The service now connects the already-merged DTO, prompt-contract, parser, and provider seams while keeping database writes out of scope.

## Requirement Anchors

- PRD: FR-2.2 for LLM classification into supported job email categories with confidence.
- PRD: FR-2.3 for typed structured extraction validated before downstream use.
- PRD: FR-2.6 for prompt and model versioning support in reproducible re-runs.
- PRD: NFR-2 for cost transparency through token accounting handoff.
- PRD: NFR-4 and NFR-5 for reliable, typed, maintainable pipeline behavior.
- Groundwork: Phase 2 classification pipeline service layer.

## How To Use

Construct `ClassificationService` with `AppSettings`, a configured `LLMProvider`, and optionally a clock for deterministic tests.
Call `classify_candidates()` with retained `EmailClassificationCandidate` values.
Accepted results are returned in `accepted` with a validated extraction and storage-ready classification.
Storage-ready classification records remain available through `classifications`.
Rejected provider outputs are returned in `malformed` with only public-safe reason metadata.
