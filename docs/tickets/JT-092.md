# JT-092 - Persist Email Classifications

## What Changed

- Added `POST /classification/run` endpoint in `backend/app/api/classification.py` that triggers the `StructuredExtractionService.run_batch()` flow.
- Added `ClassificationRunResponse` DTO in `backend/app/models/classification.py` for the public API response.
- Added `get_writable_email_repository`, `get_classification_run_repository`, and `get_structured_extraction_service` FastAPI dependencies in `backend/app/api/dependencies.py`.
- The endpoint accepts no request body; it reads retained email candidates needing classification from the database, calls the configured LLM provider through the existing `StructuredExtractionService`, and idempotently stores `email_classifications` rows and `classification_runs` accounting.
- The response returns run id, provider, model, prompt version, timestamps, candidate/classified/malformed counts, token totals, estimated cost, classification mode, and LLM provider name.

## Why

JT-092 bridges the last gap between the read-only classification planning endpoints (`GET /classification/estimate` and `GET /classification/reprocessing-plan`) and actual persistence. Users can now inspect an estimate, review the reprocessing plan, and trigger a run that idempotently stores results.

## How To Use

First review the pending work with:

```sh
curl http://127.0.0.1:8000/classification/reprocessing-plan
```

Then trigger a classification batch:

```sh
curl -X POST http://127.0.0.1:8000/classification/run
```

The response includes the run accounting summary. Each classified email gets an `email_classifications` row and the run gets a `classification_runs` row.
