# JT-094 - Structured Extraction Service

## What Changed

JT-094 adds `StructuredExtractionService` in `backend/app/services/structured_extraction.py`.
The service selects retained raw-email candidates that need the configured model and prompt version, builds the existing classification prompt request, calls the configured `LLMProvider`, validates the provider response through the Pydantic extraction parser, and stores only accepted `email_classifications` rows.
Malformed provider output returns public-safe `MalformedLLMExtraction` values and does not write classification, application, or event rows.
The service returns accepted typed extraction results for later aggregation work and records one `classification_runs` accounting row for the attempted batch.
The parser now consumes the merged prompt contract fields `application_status` and `event_type`, maps them into `JobApplicationExtraction.status` and `JobApplicationExtraction.event_type`, and keeps strict validation for malformed scalar types.

## Why

FR-2.3 requires structured extraction of application facts from retained job-search messages.
NFR-5 requires typed DTOs and provider seams rather than raw dictionaries or provider-specific payloads.
This change keeps the LLM call behind `LLMProvider`, keeps storage behind repositories, and rejects malformed output before persistence.

## How To Use

Create `StructuredExtractionService` with `AppSettings`, `EmailRepository`, `ClassificationRunRepository`, and an `LLMProvider` adapter.
Call `await service.run_batch()` to process up to `settings.classification_batch_size` retained candidates for `settings.classification_prompt_version` and the configured classification model.
Use `result.accepted_results` as the typed input to later aggregation work.
Inspect `result.malformed_results` for public-safe rejected provider output.
Read `result.run_record` or `ClassificationRunRepository.fetch_run()` for token accounting.

## Scope Notes

This ticket does not implement application aggregation, dedupe, ghost inference, or application event writes.
Those responsibilities need the later aggregation module and repository write paths.
The service intentionally persists only accepted classification rows and returns typed extraction facts to the next pipeline stage.
