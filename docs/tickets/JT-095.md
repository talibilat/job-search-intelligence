# JT-095 - Handle Malformed LLM Output

## What changed

JT-095 adds a focused classification pipeline parser for provider-neutral LLM responses.
The parser accepts only JSON objects that validate against the expected structured extraction DTO after a clean provider finish reason.
Valid responses produce an `EmailClassificationRecord` plus typed extracted application fields.
Malformed responses produce a `MalformedLLMExtraction` quarantine result instead of a classification record.
Malformed results carry only email ID, model, prompt version, reason, and a public-safe message, not raw LLM content.

## Why

Phase 2 depends on classification and extraction being trustworthy before aggregation writes applications or events.
Malformed LLM output must not silently become stored classification data.
Returning a separate malformed result keeps downstream callers from accidentally storing invalid extractions.

## How to use

Call `parse_llm_extraction_response` with an `LLMGenerationResponse`, the raw email ID, the prompt version, and the classification timestamp.
Handle `AcceptedLLMExtraction` by storing its `classification` and passing its `extraction` to later aggregation work.
Handle `MalformedLLMExtraction` by rejecting or quarantining it without writing `email_classifications`, `applications`, or `application_events` rows.
JT-094 wires accepted classification-row storage through `StructuredExtractionService`; aggregation wiring is still later Phase 2 work.
The checked-in eval runner currently exercises synthetic `LLMGenerationResponse` rows from `backend/evals/golden_set.jsonl` through the parser and reports precision, recall, and threshold pass/fail.
