# JT-096 - Track Classification Tokens And Cost

## What Changed

JT-096 adds typed local storage for one classification accounting row per run.
The new `classification_runs` table stores provider, model, prompt version, start and completion timestamps, candidate and classified counts, prompt tokens, completion tokens, total tokens, and estimated USD cost.
The new `ClassificationRunRecord` DTO validates non-negative counts, prevents classified rows from exceeding candidate rows, and rejects total token counts below the prompt plus completion token counts.
The new `ClassificationRunRepository` can upsert and fetch run records so the later classification pipeline can persist accounting without embedding SQL in service code.

## Why

FR-0.4 and NFR-2 require cost transparency for bulk classification.
This change gives classification runs a durable SQLite accounting target while preserving local-first storage and avoiding telemetry.
It does not change classification prompts, categories, model behavior, extraction behavior, or aggregation behavior.

## How To Use

Create a `ClassificationRunRecord` after a classification run completes, using the provider's reported `LLMTokenUsage` and the service's cost estimate.
Call `ClassificationRunRepository.upsert_run(record)` to store or replace the accounting row for that run ID.
Call `ClassificationRunRepository.fetch_run(run_id)` to read the stored run accounting row back as a typed DTO.

## Verification

Focused tests cover the DTO validation rules, repository upsert and fetch behavior, migration column layout, and SQLite accounting constraints.
Backend verification should run `ruff`, `mypy`, and the relevant pytest suite before merging.
The golden-set eval is not changed by this ticket because JT-096 does not alter classification prompts, models, categories, extraction schemas, or classifier decision behavior.
