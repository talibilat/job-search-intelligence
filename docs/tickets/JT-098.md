# JT-098 - Classification Mode Config

## What Changed

JT-098 adds setup and provider-config support for classification mode preselection.
The supported modes remain `hybrid`, `llm`, and `local`.
Setup status and provider config responses now expose `recommended_classification_mode` so clients can show the provider-based default separately from an explicit current choice.

## Preselection Rules

Azure OpenAI setup preselects `hybrid` because it keeps hosted model usage cost-controlled through the heuristic pre-filter.
Ollama setup preselects `local` because it preserves the local-first path when only the local provider is selected.
The `llm` mode remains accepted as an explicit user choice for runs that intentionally prefer model coverage over pre-filter cost control.

## API Behavior

`GET /setup/status` returns the current `classification_mode` and the provider-based `recommended_classification_mode`.
`POST /setup` may omit `classification_mode`, in which case the backend applies the same preselection rule before validation.
`GET /config/providers` returns the recommendation alongside the selected provider config.
`PUT /config/providers` applies the provider-based preselection when the LLM provider changes and the request does not explicitly include a classification mode.

## Verification

Backend verification should cover setup status, setup submission, provider config reads, and provider config updates.
Frontend verification should cover the setup page rendering the recommended mode as the checked radio option.
Because this ticket changes configuration and setup behavior rather than classification prompts, models, categories, or extraction schemas, the golden-set eval is not expected to change classification accuracy.
