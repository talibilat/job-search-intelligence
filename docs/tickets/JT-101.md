# JT-101 - Enforce golden set gate

## What changed

Added the golden-set classification eval as an explicit, visible gate in
the backend CI workflow, so a regression in precision or recall blocks
merges independently of the general pytest suite.

## Why

JT-100 (merged but unclosed) delivered `backend/evals/run_eval.py` with
threshold enforcement (non-zero exit code when precision < 90% or
recall < 85%). The eval was already tested through pytest
(`tests/test_golden_eval.py` calls it via subprocess with `check=True`),
so CI already enforced the gate indirectly. This ticket made the gate
explicit by adding a dedicated CI step, making it immediately visible
in the CI output when it passes or fails.

## What files changed

- `.github/workflows/backend-ci.yml` — added `Run golden-set eval` step
  before `Run pytest`
- `docs/tickets/JT-101.md` — this file

## What did not change

The eval runner itself (`backend/evals/run_eval.py`), its thresholds
(MIN_PRECISION = 0.90, MIN_RECALL = 0.85), the golden-set fixture
(`backend/evals/golden_set.jsonl`), and the existing tests were already
correct and were not modified.

## How to verify

```bash
cd backend
uv run python -m evals.run_eval
```

Expected output: `status: passed`, `pass: true`, exit code 0.

If the golden set or eval runner changes in a way that drops precision
below 0.90 or recall below 0.85, the step returns exit code 1 and CI
fails, blocking the merge.

## Relevant requirements

FR-2.7, Phase 2 gate (>= 90% precision AND >= 85% recall on job-vs-not).
