# JT-102 - Company Normalization

JT-102 adds deterministic company-name normalization for Phase 2 aggregation.

## Requirement Mapping

This change maps to `FR-2.4`, `NFR-4`, `NFR-5`, and the Phase 2 classify, extract, and aggregate deliverable.

Aggregation groups many emails into one canonical application record.
Company names extracted from email can vary by punctuation, legal suffix, casing, article, and domain-like source value.
The new helper returns a stable grouping key for those variants so downstream aggregation can compare companies without using raw extracted display names as identity.

## Behavior

`normalize_company_name(company: str) -> str` lives in `backend/app/services/normalization.py`.
It lowercases with Unicode case folding, decodes HTML entities, removes combining marks, expands ampersands to `and`, treats punctuation as separators, removes leading `the`, removes domain prefixes and suffixes for domain-like values when those suffixes are trailing terms, drops trailing plain and dotted legal suffixes, and collapses whitespace.

Examples:

```text
OpenAI, Inc. -> openai
The Example Group LLC -> example group
Johnson & Johnson -> johnson and johnson
jobs.example.com -> example
Example L.L.C. -> example
```

Blank input returns an empty key.
The helper is pure and does not touch the database, provider code, raw email content, or API responses.
Application grouping-key assembly remains in JT-104.
