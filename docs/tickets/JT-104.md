# JT-104 - Application Grouping Key

## What Changed

JT-104 adds `ApplicationGroupingKey` and `build_application_grouping_key()` in `backend/app/pipeline/aggregate.py`.
The helper combines normalized company, normalized role, and either an opaque provider thread ID or a deterministic time-window bucket into one typed key for Phase 2 aggregation.

## Requirement Mapping

This change maps to `FR-2.4`, `NFR-4`, `NFR-5`, and the Phase 2 classify, extract, and aggregate deliverable.
It supports the invariant that one real application can have many emails and should become one `applications` row with multiple `application_events`.

## Behavior

`build_application_grouping_key()` calls the already-merged `normalize_company_name()` and `normalize_role_title()` helpers instead of reimplementing normalization.
It trims but does not case-fold provider thread IDs because thread IDs are opaque provider-owned values.
When a non-blank thread ID is present, the key stores that thread signal and leaves `time_window_start` as `None` so a later response in the same thread is not split by date.
When the thread signal is missing, it converts timezone-aware datetimes to UTC before computing the time-window date.
It treats blank company, role, thread, or timestamp values as explicit `None` signals instead of collapsing them into misleading empty strings.
The default grouping window is 30 days and can be overridden with a positive `window_days` value.
Non-positive window values raise `ValueError`.
The frozen Pydantic DTO stores `(normalized_company, normalized_role, thread_id, time_window_start, time_window_days)` and exposes `as_tuple()` for deterministic dictionary or set keys without introducing database schema or API changes.

## How To Verify

Run the focused aggregation test from the repository root:

```sh
uv run --project backend pytest backend/tests/test_application_grouping_key.py -v
```

Run the related normalization tests because this key calls the JT-102 and JT-103 helpers:

```sh
uv run --project backend pytest backend/tests/test_company_normalization.py backend/tests/test_role_normalization.py -v
```

Run backend quality checks for the touched Python files:

```sh
uv run --project backend ruff check backend/app/pipeline/aggregate.py backend/app/pipeline/__init__.py backend/tests/test_application_grouping_key.py
uv run --project backend ruff format --check backend/app/pipeline/aggregate.py backend/app/pipeline/__init__.py backend/tests/test_application_grouping_key.py
```

Run mypy from `backend/` so it reads the backend project configuration:

```sh
uv run mypy
```
