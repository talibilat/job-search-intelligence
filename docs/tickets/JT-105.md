# JT-105 - Implement application upsert

## What changed

Created the aggregation service that groups classification extraction results by
`ApplicationGroupingKey` and upserts one application per group with a timeline
of events.

## Files touched

- **`backend/app/pipeline/aggregate.py`** - Added `make_application_id()` and
  `make_event_id()` which generate deterministic UUIDv5 identifiers from
  grouping key / event identity signals. This is the core of idempotency:
  the same key always produces the same IDs, so `ON CONFLICT(id) DO UPDATE`
  prevents duplicates.
- **`backend/app/db/repositories/application.py`** - Added
  `upsert_application()` method that inserts or updates an application row
  using `ON CONFLICT(id) DO UPDATE`.
- **`backend/app/db/repositories/event.py`** - Added `upsert_event()` method
  that inserts or updates an event row using `ON CONFLICT(id) DO UPDATE`.
- **`backend/app/db/repositories/email.py`** - Added `get_thread_id()` helper
  so the aggregation service can look up thread IDs from raw email rows.
- **`backend/app/services/aggregation.py`** - New file. `AggregationService`
  takes `AcceptedLLMExtraction` results, filters to job-related ones, enriches
  them with thread IDs, groups by `ApplicationGroupingKey`, and upserts one
  application + N events per group. Merges extracted fields (tech stack,
  salary range, etc.) across emails in the same group. Derives the current
  application status from the highest-priority status seen across all events.
- **`backend/tests/test_aggregation_service.py`** - 8 tests covering single
  extraction, multiple extractions grouped into one application, idempotency
  (re-running produces no duplicates), skip of non-job-related results,
  separate grouping keys, incremental updates, tech stack merging, and empty
  input.

## Design decisions

- **Deterministic IDs instead of a schema migration.** Application IDs are
  UUIDv5 derived from the canonical JSON of the grouping key. Event IDs are
  UUIDv5 derived from `(application_id, email_id, event_type, event_at)`.
  This avoids adding a new column or index to the database schema while
  keeping upserts idempotent via `ON CONFLICT(id)`.
- **Thread IDs looked up at aggregation time.** Thread IDs are not passed
  through the classification pipeline. The aggregation service looks them up
  from `raw_emails` by email ID, which is a simple and non-invasive approach.
- **Status derivation by priority.** When multiple emails in the same
  application group have different statuses, the highest-priority status
  wins (offer > rejected > interview > assessment > in_review > applied).
- **Source defaults to "other".** The classification prompt does not extract
  an application source field, so it defaults to "other" for the first pass.
  This can be refined in a later ticket.
