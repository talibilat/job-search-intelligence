# JT-106 - Implement event timeline upsert

JT-106 completes idempotent application event timeline upserts for Phase 2 aggregation.

## Requirement Mapping

This change maps to `FR-2.4`, `NFR-4`, `NFR-5`, and the Phase 2 classify, extract, and aggregate deliverable.
It protects the invariant that `applications` plus `application_events` are the factual source of truth for job-search history.

## Behavior

Aggregation now upserts `application_events` through deterministic event IDs derived from application, email, event type, and event timestamp signals.
When structured extraction omits `event_at`, the timeline uses the source raw email `sent_at` timestamp before falling back to the stored classification timestamp.
That keeps repeated aggregation runs from creating duplicate events for the same classified email.

## How To Verify

Run the focused aggregation test from the repository root:

```sh
uv run --project backend pytest backend/tests/test_aggregation_service.py -q
```

Run backend quality checks for the touched files:

```sh
uv run --project backend ruff check backend/app/services/aggregation.py backend/app/db/repositories/email.py backend/tests/test_aggregation_service.py
uv run --project backend ruff format --check backend/app/services/aggregation.py backend/app/db/repositories/email.py backend/tests/test_aggregation_service.py
```

Run focused mypy from `backend/` so it reads the backend project configuration:

```sh
uv run mypy app/services/aggregation.py app/db/repositories/email.py tests/test_aggregation_service.py
```
