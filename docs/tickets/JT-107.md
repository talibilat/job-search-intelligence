# JT-107 - Implement Status Derivation

## What Changed

Application aggregation now derives `current_status` from the ordered event timeline for an application group.
Events are ordered by the extracted event timestamp, then the raw email `sent_at`, then the classification timestamp.
The latest status-bearing event wins, so a later rejection can override an earlier offer instead of losing to a static priority list.
Aggregation combines already-persisted `application_events` with the current batch before deriving status, so incremental runs do not regress status when older evidence arrives late.

## Why

FR-2 requires the `applications` table and `application_events` timeline to be the factual source of truth.
JT-105 used a highest-priority status rule as an initial slice.
JT-107 replaces that with chronological derivation so `current_status` reflects the actual event history.

## Behavior

Status derivation maps event types to application statuses.
`applied` maps to `applied`.
`response` maps to `in_review`.
`assessment` maps to `assessment`.
`interview_scheduled` maps to `interview`.
`rejection` maps to `rejected`.
`offer` maps to `offer`.
`ghost_inferred` maps to `ghosted`.
`feedback` preserves the current status unless extraction supplied an explicit status.

Aggregation also uses the same deterministic event timestamp fallback when upserting event rows.
This keeps event IDs stable when the LLM extraction omits `event_at`.
The same fallback now feeds no-thread grouping keys, so missing LLM event dates still use the email timestamp for the 30-day grouping window.
If an existing application has `manual_lock` set, automatic aggregation preserves its locked `current_status` instead of overwriting a user correction.
