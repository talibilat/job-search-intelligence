# JT-108 - Implement Ghost Inference

## What Changed

Added `POST /applications/ghost-inference` to run ghost inference against the local SQLite application timeline.
The endpoint uses `AppSettings.ghost_threshold_days`, which defaults to 30 and can be overridden with `JOBTRACKER_GHOST_THRESHOLD_DAYS`.
The runner selects applied applications whose timeline has an `applied` event, has no response-like event, has no existing `ghost_inferred` event, and is past the configured silence threshold.
For each eligible application, it writes one deterministic `ghost_inferred` event with `email_id = null` at `last_activity_at + threshold_days`.
After writing the event, it updates the application summary by replaying the event timeline through JT-107 status derivation, so `current_status` becomes `ghosted` from the factual timeline.
Before adding new ghost events, the runner reconciles existing `ghost_inferred` events against the current timeline and threshold.
It removes stale inferred ghost events when response evidence exists, when the configured threshold makes the ghost date no longer due, or when the inferred event date no longer matches the configured threshold.
If a threshold change still leaves the application overdue, the same run retracts the old inferred event and writes a new deterministic one at the corrected date.
Manual-locked applications are not overwritten.
They are returned in `manual_conflict_application_ids` so user corrections remain visible instead of silently changed.

## Why

FR-2.5 requires applications to become `ghosted` when no response arrives after a tunable threshold.
The implementation preserves the product invariant that factual answers come from `applications` plus `application_events`.
Persisting a `ghost_inferred` timeline event also keeps later dashboard and metrics work deterministic.

## How To Use

Run ghost inference with:

```bash
curl -X POST http://127.0.0.1:8000/applications/ghost-inference
```

The response includes the evaluated threshold, the IDs that were newly marked ghosted, and any manual-lock conflicts.
It also includes `ghost_events_retracted` and `retracted_application_ids` when stale inferred events are removed or recomputed.
Calling the endpoint again is idempotent for the same database state and does not create duplicate ghost events.
