# JT-109 - Reprocessing Version Controls

## What Changed

JT-109 adds `GET /classification/reprocessing-plan`.
The endpoint reports how many retained classification candidates are already current for the configured target model and prompt version, and how many need first-time classification or version-controlled reprocessing.
The route is read-only and does not call an LLM.
It does not expose email content.
It does not create a missing SQLite database during a read.

## Why

FR-2.6 requires reproducible re-runs where unchanged mail produces no work and prompt or model changes select only stale rows.
The reprocessing plan makes that selection policy visible before a later classification runner executes it.
This preserves local-first deterministic behavior and keeps dashboard truth grounded in stored classification rows rather than LLM guesses.

## Selection Policy

The target model comes from the configured LLM provider settings.
For Azure OpenAI this is `azure_openai_chat_deployment`.
For Ollama this is `ollama_chat_model`.
The target prompt version comes from `classification_prompt_version`.
Retained candidate emails are selected for reprocessing when they have no stored classification, a stored `model` different from the target model, or a stored `prompt_version` different from the target prompt version.
The returned stale buckets are mutually exclusive: model mismatches are counted in `stale_model_count`, and prompt mismatches are counted in `stale_prompt_version_count` only when the stored model already matches the target.
