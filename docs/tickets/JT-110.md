# JT-110 - Aggregation Idempotency Tests

## What Changed

Added focused backend tests that prove aggregation reruns do not duplicate canonical `applications` rows or `application_events` timeline rows.

The tests cover a full rerun of the same two-email application evidence and an incremental replay where previously processed evidence is included again alongside newer evidence.

## Why

FR-2.4 and FR-2.6 require aggregation to reconstruct one application from many classified emails and make unchanged reruns deterministic.

NFR-4 requires the sync and aggregation pipeline to be idempotent so repeated processing does not inflate dashboard counts or timelines.

## Verification

Run focused test and lint verification from the repository root:

```sh
uv run --project backend pytest backend/tests/test_aggregation_service.py -q
uv run --project backend ruff check backend
```

Run mypy from `backend/` because its configured targets are relative to that directory:

```sh
uv run mypy
```
