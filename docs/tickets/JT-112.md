# JT-112 - Manual Split API

## What Changed

JT-112 adds `POST /applications/{application_id}/split`.
The endpoint moves selected existing `application_events` from an incorrectly grouped source application into a new manually locked application.
It returns the updated source application, the new application, the moved events, and the audited `split` correction row.

## Why

Manual split corrections protect the product invariant that factual job-search answers come from one clean `applications` table and its event timeline.
They let a user repair over-grouped applications without losing event evidence or making dashboard breakdown fields ambiguous.

## How It Works

`backend/app/api/applications.py` keeps the route thin and delegates split behavior to `ApplicationCorrectionService`.
The service requires the application, event, and correction repositories to share one SQLite connection so the event move, source summary update, new application insert, and correction audit row commit atomically.
The request must include at least one unique selected event ID, may include corrected source segmentation fields, and must include the new application's factual segmentation fields.
The split must leave at least one event on the source application.

The new application ID is deterministic for the source application and moved event IDs, using the `manual-split-*` prefix.
Both resulting applications are manually locked.
The source application keeps its locked status when it was already manually locked; otherwise, source and target statuses are derived deterministically from their remaining event timelines.
First-seen and last-activity timestamps are recalculated from each resulting timeline.
Source and target segmentation fields such as company, role, source, salary, location, work mode, seniority, sponsorship, and tech stack are preserved for deterministic dashboard breakdowns.

The API returns a typed `404` when the source application does not exist.
It returns a typed `409` when selected events are missing from the source application, the split would move every source event, the deterministic target application already exists, or the repositories are not sharing one SQLite connection.

## Verification

Run `uv run pytest tests/test_application_split_api.py -v` from `backend/` to verify event reassignment, audit correction recording, source and target summary recalculation, locked status handling, segmentation-field preservation, conflict handling, and OpenAPI schema coverage.
