# JT-113 - Manual Status And Event Edit API

## What Changed

JT-113 adds audited manual correction endpoints for application status and timeline event edits.

The backend now exposes `PATCH /applications/{application_id}/status` for correcting `applications.current_status`.

The backend now exposes `PATCH /applications/{application_id}/events/{event_id}` for correcting one `application_events` timeline row.

Both endpoints return the updated record and the `application_corrections` audit row that was created.

## Why

FR-2.8 requires manual correction support for aggregation mistakes.

The source of truth remains the local SQLite `applications` row plus its `application_events` timeline.

Manual corrections set `manual_lock` on the application so automatic aggregation cannot silently overwrite the user correction.

Event edits are also protected by the `event_edit` audit record, so a later rerun that disagrees with the edited event reports a manual conflict instead of replacing the edited timeline row.

## How To Use

Send a status correction with a new status and optional reason.

```http
PATCH /applications/app-1/status
Content-Type: application/json

{
  "current_status": "rejected",
  "reason": "The rejection email was missed by extraction."
}
```

Send an event correction with at least one event field and an optional reason.
The editable event fields are `event_type`, `event_at`, `email_id`, and `extract_note`.

```http
PATCH /applications/app-1/events/event-1
Content-Type: application/json

{
  "event_type": "interview_scheduled",
  "event_at": "2026-07-07T14:00:00Z",
  "extract_note": "Recruiter scheduled a phone screen.",
  "reason": "The timeline should show the interview invite."
}
```

For `ghost_inferred` events, `email_id` must be `null`.

For all other event types, `email_id` must point to the source email.

When changing `email_id`, the new non-null source email must already exist in `raw_emails`.

Send explicit JSON `null` for `extract_note` when the note should be cleared.

Send explicit JSON `null` for `email_id` only when changing the event to `ghost_inferred`.

Changing `email_id`, `event_type`, or `event_at` can change the deterministic event ID returned in the response.

Event edits that do not change any event data are rejected.

After an event edit, the application status and last activity are replayed from the edited timeline.

## Verification

Relevant tests cover status edits, event edits, audit rows, `manual_lock`, source-email validation, no-op rejection, event ID regeneration, status replay, stale extracted-status clearing, event conflict protection, typed API errors, and OpenAPI response documentation.

Run the focused checks from `backend/` with:

```sh
uv run pytest tests/test_manual_application_edit_service.py tests/test_manual_application_edit_api.py tests/test_aggregation_service.py
```
