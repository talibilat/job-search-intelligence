# JT-114 - Correction Locks

## What Changed

JT-114 adds an audited reset path for manual correction locks.

The backend now exposes `POST /applications/{application_id}/reset-lock`.

The endpoint clears `applications.manual_lock` on one locked application and writes a `reset_lock` row to `application_corrections`.

The reset response returns the updated application and the audit correction record.

## Why

FR-2.8 requires user-corrected grouping and status to be locked from automatic overwrite by default.

The existing merge, split, status edit, and event edit paths already set `manual_lock` when a user corrects an application.

This slice completes the lock lifecycle by giving the user an explicit audited reset when they want automatic aggregation to manage the application summary again.

## How To Use

Send a reset request with an optional reason.

```http
POST /applications/app-1/reset-lock
Content-Type: application/json

{
  "reason": "Let aggregation manage this application again."
}
```

The endpoint returns `404` when the application does not exist.

The endpoint returns `409` when the application is not currently manually locked.
