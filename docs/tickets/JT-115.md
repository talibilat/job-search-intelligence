# JT-115 - Correction Conflict Detection

## What changed

JT-115 adds durable correction conflict records for automatic evidence that disagrees with a manually locked application or manually edited event.
Aggregation still refuses to overwrite locked correction state, and now records the rejected proposed evidence in `application_correction_conflicts`.
The application API exposes those records at `GET /applications/{id}/correction-conflicts` so a UI or operator can inspect what new evidence needs manual review.

## Why

FR-2.8 requires user-corrected grouping and status to be locked by default while surfacing conflicts when new evidence disagrees.
The previous merged behavior returned conflict counts and application IDs from aggregation, but those details disappeared after the run.
Persisting conflict records keeps the local SQLite database as the source of truth and gives later UI work a stable read path.
