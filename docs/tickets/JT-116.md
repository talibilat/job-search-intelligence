# JT-116 - Application Detail API

## What Changed

JT-116 adds `GET /applications/{id}`.
The endpoint returns one canonical `ApplicationRecord` from the local SQLite `applications` table.
It returns the standard typed `404` API error when no application row matches the requested ID.

## Why

Application detail reads preserve the product invariant that factual job-search answers come from the clean `applications` table.
The route exposes that source of truth without involving an LLM, generating counts, or reading email content.

## How It Works

`backend/app/api/applications.py` keeps the route thin and delegates to `ApplicationDetailService`.
The service reads through `ApplicationRepository.get_by_id()` and raises `ApplicationNotFoundError` for missing rows.
The API maps that service error to the standard public `not_found` response shape.
The OpenAPI schema and generated frontend client include the `ApplicationRecord` response and typed 404 error.

## Verification

Run `uv run --project backend pytest backend/tests/test_application_detail_api.py -v` to verify the application detail route, typed not-found response, OpenAPI schema, repository lookup, and missing-table behavior.
