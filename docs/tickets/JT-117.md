# JT-117 - Application Events API

## What Changed

JT-117 adds `GET /applications/{id}/events`.
The endpoint returns the canonical event timeline for one application from local SQLite.
Events are ordered by `event_at` and then `id`, matching the existing repository timeline ordering.
An existing application with no event rows returns an empty list.
Missing applications return the standard typed `404` API error.

## Why

The product invariant is that factual job-search answers come from the `applications` table and its event timeline.
This API exposes the `application_events` half of that invariant without using an LLM, producing dashboard counts, or reading private email bodies.

## How It Works

`backend/app/api/applications.py` keeps the route thin and delegates to `ApplicationEventsService`.
The service verifies the parent application exists through `ApplicationRepository.get_by_id()` before reading timeline rows through `EventRepository.list_by_application_id()`.
The API maps missing parent applications to the standard public `not_found` response shape.
The response model is `list[ApplicationEventRecord]`, so OpenAPI and the generated frontend client can use the typed event DTO.
