# JT-118 - Applications List API

## What Changed

`GET /applications` returns canonical application rows from the local SQLite `applications` table.
The response uses the existing `ApplicationRecord` DTO, so it preserves the same boundary contract as `GET /applications/{id}`.

## Filters

The endpoint supports these optional query parameters:

- `status`
- `source`
- `sponsorship`
- `first_seen_from`
- `first_seen_to`
- `role`
- `salary_min`
- `salary_max`
- `work_mode`

`first_seen_from` and `first_seen_to` filter the canonical `first_seen_at` application timestamp.
`role` performs a case-insensitive substring match against `role_title`.
`salary_min` and `salary_max` filter applications whose stored salary range overlaps the requested band.

## Why

This implements the base read API needed for Phase 2 applications inspection and later dashboard filters.
The route stays thin, service code owns the API-facing filter mapping, and repository code owns the deterministic SQLite query.
