# JT-119 - Application Correction UI

## What Changed

JT-119 adds an application detail screen for manual correction workflows.

The frontend now handles `/applications/{application_id}` routes and fetches the application detail plus its event timeline from the generated API client.

The detail screen exposes correction tools for manual status edits, timeline event edits, duplicate merges, and event splits.

Each correction posts through the existing audited backend endpoints and shows a success or public-safe error message.

## Why

FR-2.8 requires a manual correction path for aggregation mistakes.

The correction UI keeps the local SQLite `applications` row and `application_events` timeline as the source of truth.

The UI does not infer counts, call an LLM, log private email content, or bypass backend audit behavior.

## How To Use

Open an application detail route such as `/applications/app-1`.

Use **Edit current status** to post a new `applications.current_status` value with an optional reason.

Use **Edit timeline event** to correct an event type, timestamp, source email, or note for a selected event.

Use **Merge duplicate application** to move a source application into the current target application by source application ID.

Use **Split selected events** to move selected timeline events into a new application with a company and role.

## Verification

Focused frontend coverage exercises the detail route, application and event loading, status correction, event correction, merge correction, split correction, generated-client request bodies, and success messaging.

Run the focused check from `frontend/` with:

```sh
npm run test -- src/App.test.tsx
```

Run the full frontend gate from `frontend/` with:

```sh
npm run check
```
