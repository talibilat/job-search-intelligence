# JT-122 - Metrics Repository Queries

## What Changed

Expanded `MetricsRepository` with deterministic query methods for Phase 3 metrics.

The repository now exposes total application counts, rate metrics, funnel stages, application-volume timeseries, and breakdown rows for planned dashboard dimensions.

Added typed Pydantic DTOs for rate, funnel, timeseries, and breakdown query results.

Added focused repository tests that seed a local SQLite database and verify the metric math against synthetic applications and event timelines.

No new `/metrics/*` endpoints or frontend calls were added; later tickets own service, route, OpenAPI, and dashboard exposure.

Rate rows cover response, rejection, ghost, application-to-interview, and interview-to-offer metrics.

Ghost rate uses the caller-provided ghost cutoff instead of only the persisted `ghosted` status.

Interview-to-offer metrics count offers only when the offer event occurs after an interview event in the deterministic event ordering.

Funnel stages currently cover applied, response, assessment, interview, and post-interview offer counts.

The application-volume timeseries groups applications by the UTC date prefix of `applications.first_seen_at`.

Breakdown rows cover `role`, `source`, `salary`, `tech`, `sponsorship`, `seniority`, and `work_mode`; tech rows normalize and deduplicate stack values per application.

## Why

JT-122 prepares the repository layer for the later `/metrics/rates`, `/metrics/funnel`, `/metrics/timeseries`, and `/metrics/breakdown` endpoints.

The queries keep dashboard metric math deterministic and tied to the canonical `applications` table plus `application_events` timeline.

The repository methods keep quantitative math outside LLM paths and prepare a typed seam for later metrics services and routes.
