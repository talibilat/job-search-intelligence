# JT-145 - Q-10 Live Applications

## What Changed

The dashboard now answers Q-10 by showing applications that are still live and awaiting a response.
The panel loads deterministic `ApplicationRecord` rows from the existing `GET /applications` endpoint for the active statuses `applied`, `in_review`, `assessment`, and `interview`.
It does not introduce or extend the metrics summary endpoint, because Q-10 can be answered from the existing canonical applications API without duplicating sibling metrics work.

## Why

Q-10 asks which applications are still live right now.
Using the canonical applications API keeps the answer grounded in the `applications` table and the event-derived `current_status` field.
Terminal statuses such as `offer`, `rejected`, `ghosted`, and `withdrawn` are excluded from the live list.

## How To Verify

Open `/dashboard` with the local backend running against a database that contains application rows.
The Q-10 panel should show a count and list of live applications ordered by oldest `last_activity_at` first, so the longest-waiting applications appear first.
Run `npm run test -- App.test.tsx` from `frontend/` to verify the dashboard renders live applications from deterministic application API responses.
