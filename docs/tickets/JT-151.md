# JT-151 - Q-16 Full Funnel

## What Changed

Added `GET /metrics/funnel` for Q-16.

The endpoint returns deterministic funnel stage counts from local SQLite `applications` and `application_events` data.

The response shape is `MetricsFunnelResponse` with ordered `MetricFunnelStage` rows.

The dashboard now renders a Q-16 full funnel section from the generated frontend API client.

The rendered stages are applied, screen, interview, final, and offer.

Screen counts applications with explicit `response` events.

Final is included in the response contract and remains `0` until final-round evidence exists in the event taxonomy.

Offers continue to use the existing repository rule that counts offers only after interview evidence.

The endpoint accepts the shared dashboard metric filters so the funnel stays consistent with filtered dashboard views.

## Why

JT-151 covers Phase 3, FR-3.2, NFR-5, NFR-7, and Q-16.

Q-16 needs a dashboard-visible funnel that reconciles with the same deterministic source of truth as the other Phase 3 metrics.

The implementation keeps the route thin, puts behavior in the metrics service and repository seam, and avoids any LLM-derived counts.

## Usage Notes

Read Q-16 from `GET /metrics/funnel`.

Use the ordered `stages` list for dashboard cards or charts.

Pass the same dashboard filter query params as `/metrics/breakdown` when showing a filtered dashboard view.

Do not treat screen or final as new persisted event types.

Screen is a deterministic metric stage derived from the existing event contract.

Final is present as a stable stage but is not inferred from rejection or offer events.
