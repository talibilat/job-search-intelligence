# JT-152 - Q-17 Time To First Response

## What Changed

Extended `GET /metrics/summary` with an `average_time_to_first_response` metric for Q-17.

The response includes `average_time_to_first_response.application_count` and `average_time_to_first_response.average_hours` so the dashboard value can be reconciled to deterministic counts and duration math.

The metric measures from `applications.first_seen_at` to the first response-like timeline event on each application.

Response-like events use the existing `RESPONSE_LIKE_APPLICATION_EVENT_TYPES` contract and include response, assessment, interview, feedback, rejection, and offer evidence.

Applications without response evidence are excluded from the average.

The dashboard now renders an Avg time to first response card from the deterministic `/metrics/summary` endpoint.

When no applications have response evidence, `average_hours` is `null` because there is no meaningful average.

## Why

JT-152 covers Phase 3, FR-3.2, NFR-5, NFR-7, and Q-17.

The implementation keeps the dashboard truth grounded in local SQLite `applications` plus `application_events` data and avoids any LLM-derived timing or count.

## Usage Notes

Read Q-17 from `GET /metrics/summary` at `average_time_to_first_response`.

Use the included application count when showing the duration so users can trace the displayed average back to deterministic response-evidence rows.
