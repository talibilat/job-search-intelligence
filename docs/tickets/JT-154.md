# JT-154 - Q-19 Personal Ghost Threshold

## What Changed

Extended `GET /metrics/summary` with a `personal_ghost_threshold` metric for Q-19.
The response includes an inferred threshold in days, the source of that threshold, the response sample size, the number of silent applications, and a silence-age distribution.

The threshold is inferred from historical first-response timings when response evidence exists.
When no response evidence exists, the metric falls back to the configured `ghost_threshold_days` value.

The dashboard now renders a Personal ghost threshold section from the deterministic `/metrics/summary` endpoint.
The section shows the threshold and silence-age buckets for currently silent applications.

## Why

JT-154 covers Phase 3, FR-3.2, NFR-5, NFR-7, and Q-19.
Q-19 asks when an application is effectively dead after silence.

The implementation keeps the answer grounded in local SQLite `applications` plus `application_events` data and avoids LLM-derived counts or timing.

## Usage Notes

Read Q-19 from `GET /metrics/summary` at `personal_ghost_threshold`.
The metric composes with the shared dashboard filter query fields.

Use `threshold_source` to distinguish a threshold inferred from response timing from the configured fallback.
Use `silence_age_distribution` to show how many silent applications sit in each silence-age bucket.
