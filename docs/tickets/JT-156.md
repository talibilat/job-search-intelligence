# JT-156 - Response Rate Trend

## What Changed

The backend now exposes `GET /metrics/response-rate-trend` for Q-21.
The endpoint returns deterministic response-rate trend points with `period_start`, `application_count`, `response_count`, and `response_rate`.
The dashboard now renders a Q-21 "Response rate trend" section using that endpoint.
The request uses the shared route-backed dashboard filters so the trend composes with status, source, sponsorship, date, role, salary, and work-mode filters.

## Why

Q-21 asks whether response rate is improving over time.
The answer must come from deterministic local SQLite reads over `applications` and response-like `application_events`, not from an LLM.

## Verification

Run focused backend verification from the repository root:

```sh
uv run --project backend pytest backend/tests/test_metrics_response_rate_trend_api.py -q
uv run --project backend ruff check backend
```

Run backend type checking from `backend/`:

```sh
uv run mypy
```

Run frontend verification from `frontend/`:

```sh
npm run test -- src/pages/DashboardPage.test.tsx
npm run check
```

## Usage Notes

Start the local backend and open the dashboard.
The "Response rate trend" section answers Q-21 from `/metrics/response-rate-trend`.
If no applications exist for the active filters, the section shows an empty state rather than implying a trend.
