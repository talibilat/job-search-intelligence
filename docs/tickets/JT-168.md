# JT-168 - Diagnostics Service Foundation

## What Changed

Added typed diagnostics DTOs and a backend `DiagnosticsService` foundation.
The service builds deterministic segment comparisons from existing metrics breakdown rows.
It computes response, interview, and offer rates for each segment plus response-rate lift against the overall baseline.
It also returns strongest and weakest response-rate segments for later diagnostics API and UI tickets.

## Why It Changed

Phase 3.5 diagnostics need deterministic and light-statistics logic outside FastAPI routes.
This foundation keeps diagnostic calculations grounded in the local SQLite `applications` and `application_events` source of truth through the existing `MetricsRepository`.
It does not introduce LLM-generated counts, telemetry, schema changes, or a new endpoint.

## Linked Requirements

FR-3.4, FR-3.6, Phase 3.5, and Tier 4.

## Usage Notes

Instantiate `DiagnosticsService` with a `MetricsRepository` and call `get_diagnostics()`.
JT-169 can expose this service through `GET /metrics/diagnostics` without moving diagnostic business logic into the route handler.
