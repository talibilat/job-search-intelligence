# JT-180 - Diagnostics Fixture Tests

## What Changed

Added a private-data-free synthetic fixture for Phase 3.5 diagnostics.
Added backend tests that load the fixture through `SyntheticFixtureRepository` and verify deterministic diagnostic outputs from `DiagnosticsService`.
The tests cover unfiltered source segment comparisons and filtered LinkedIn-only diagnostics.

## Why It Changed

JT-180 asks diagnostics to be verified against synthetic data instead of relying only on inline database seeding.
The fixture keeps the `applications` and `application_events` source-of-truth invariant visible in review and reusable by later diagnostics or smoke checks.

## Verification

- `uv run --project backend pytest backend/tests/test_diagnostics_fixture.py -q`
- `uv run --project backend pytest backend/tests/test_diagnostics_fixture.py backend/tests/test_diagnostics_service.py backend/tests/test_metrics_diagnostics_api.py -q`
- `uv run --project backend ruff check backend/tests/test_diagnostics_fixture.py`
- `uv run --project backend ruff format --check backend/tests/test_diagnostics_fixture.py`
- From `backend/`: `uv run --project backend mypy tests/test_diagnostics_fixture.py`

## Usage Notes

The fixture lives at `backend/tests/fixtures/synthetic/diagnostic_job_search.json`.
Load it with `SyntheticFixtureRepository(connection).load_file(path)` before constructing `DiagnosticsService` with the same SQLite connection.
