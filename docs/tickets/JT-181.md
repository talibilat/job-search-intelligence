# JT-181 - Insights Repository

## What Changed

`InsightRepository` now persists generated cached insight rows through a typed repository API.
It saves narrative content, the source-input hash, the LLM model name, the generated timestamp, and the stale flag in the existing local SQLite `insights` table.

## Cache Behavior

`save_generated_insight()` stores the latest generated row for an insight type.
If a row for that type already exists, the repository updates it instead of creating duplicate current cache rows.
`get_cached_insight()` returns a non-stale cache hit only when the insight type, input hash, and model all match.
`mark_stale_except_inputs_hash()` marks rows stale when their stored input hash no longer matches the caller's current prepared insight input hash, including prompt version.

## Why

This is Phase 4 groundwork for FR-4 cached narrative insights.
The repository keeps database access behind the repository layer and leaves LLM synthesis, deterministic fact preparation, citations, API routes, and frontend rendering to later insights tickets.
The implementation does not let the LLM produce authoritative counts or raw SQL.
