# JT-182 - Insight Input Builder

## What Changed

`InsightInputBuilder` prepares deterministic input packages for cached narrative insights before any LLM synthesis runs.
The builder returns typed facts, cited application and email evidence, and a stable `inputs_hash` for later cache and staleness workflows.
Current insight generation includes `INSIGHT_GENERATION_PROMPT_VERSION` in that hash so prompt-template changes do not reuse legacy cached content.

## Behavior

The builder reads from the local SQLite source of truth through `InsightRepository`.
Prepared facts include total applications, application status counts, source counts, sponsorship counts, work-mode counts, and event-type counts.
Prepared evidence includes application IDs, event IDs, email IDs, application segmentation fields, event notes, retained email metadata, and retained email body text for later provider-scoped synthesis.
Retained email body text is kept out of DTO `repr` output to reduce accidental logging risk.
Debugging-only retained bodies are not copied into insight evidence body text.
The builder scopes rejection-focused inputs to rejection and feedback events so generic application confirmations cannot crowd out stronger citations.
The builder scopes weekly-action inputs to current open applications and orders that evidence newest first.
The builder includes a source fingerprint in the hashed package so source evidence changes can invalidate the input hash even when they occur outside the returned evidence limit.
It does not call an LLM, write cached insights, expose a public API, or implement regeneration.

## Why

FR-4 allows LLM narrative insights only after deterministic facts and cited source data are prepared.
This slice gives later Phase 4 generation and grounding tickets a typed seam that keeps the LLM away from authoritative counts.
