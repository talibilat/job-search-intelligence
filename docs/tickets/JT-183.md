# JT-183 - Insight Staleness Detection

## Goal

Mark cached insights stale when the local SQLite source of truth for insight inputs changes.
This implements the FR-4 cache staleness requirement without calling an LLM or generating new insight content.

## What Changed

Added an Alembic migration that creates staleness triggers for `applications`, `application_events`, and cited `raw_emails` evidence.
Fresh cached insight rows now become stale after substantive inserts, updates, or deletes in either application or event source table.
Fresh cached insight rows also become stale when retained raw email evidence fields used by citations change for an email referenced by an application event.
Existing cache lookup behavior still filters stale rows through `InsightRepository.get_cached_insight`.
No public API, frontend route, LLM provider path, prompt, or regeneration behavior changed in this ticket.

## Design

The implementation uses SQLite triggers on `applications`, `application_events`, and raw emails that are cited by application events.
Those tables provide the facts and evidence consumed by `InsightInputBuilder`, so database-level invalidation covers aggregation runs, ghost inference, manual edits, manual merges, splits, lock resets, retained evidence updates, and direct repository writes.
The triggers mark currently fresh rows in `insights` as stale after substantive inserts, updates, or deletes in each watched source table.
Application update triggers ignore metadata-only timestamp churn in `created_at` and `updated_at` so idempotent aggregation replays do not make a cached insight stale when no source facts changed.
Raw email update triggers ignore fields that are not part of insight citation evidence, such as labels, so unrelated sync metadata churn does not stale narrative caches.

## Alternatives Considered

Service-level invalidation would be explicit but would require every current and future application or event writer to remember to call it.
Repository write hooks would centralize normal application code paths but would still miss direct SQL changes and would be harder to keep value-aware without duplicating comparison logic.
SQLite triggers keep the cache invariant close to the local-first storage layer and require no public API or provider changes.

## Implementation Plan

1. Add failing migration tests that prove source inserts, substantive updates, and deletes stale cached insights.
2. Add a failing test that proves retained raw email evidence updates stale cached insights when the email is cited by an application event.
3. Add a failing test that proves idempotent `updated_at`-only application updates do not stale insights.
4. Add an Alembic migration that creates value-aware staleness triggers for source application, event, and cited raw email evidence changes.
5. Re-run focused tests, backend lint, mypy, and the relevant backend suite.
6. Rebase on latest `origin/main`, re-run verification, commit, and validate through no-mistakes unless the measured diff meets the small-change exception.
