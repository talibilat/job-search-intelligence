# JT-184 - Implement Insight Generation Service

## What Changed

JT-184 adds `InsightGenerationService` in `backend/app/services/insights_service.py`.
The service builds deterministic cited inputs through `InsightInputBuilder`, checks the fresh insight cache by type, inputs hash, and configured chat model, and calls the injected `LLMProvider` only when regeneration is needed.
Fresh generations are saved through `InsightRepository.save_generated_insight` so later API and UI slices can read the cached narrative from the existing `insights` table.

## Grounding And Safety

The service sends only the prepared `InsightInput` payload to the configured provider path.
The system prompt tells the LLM to use deterministic facts as the only numeric source, never produce authoritative counts, never emit raw SQL, and cite only the provided `citation_id` values.
JT-184 performed basic provider response validation by rejecting incomplete or blank generations.
JT-193 later hardened this path so generated insights must include citations and every cited ID must come from the prepared evidence package or role-fit outcome summaries.

## Cache Behavior

`generate_insight(type)` returns a fresh cached row without an LLM call when the stored insight matches the current input hash and configured chat model.
`generate_insight(type, force=True)` bypasses that cache and writes a new generated insight for the same type.
