# JT-187 - Add Insights Frontend Page

## What Changed

The `/insights` frontend route now renders the full Tier 5 cached insights surface for Q-40 through Q-46.
The page loads cached insight records from `GET /insights`, shows a cache summary, renders stale cache state, displays parsed citation IDs from grounded insight content, links application citations to application detail pages, and offers a per-insight regenerate action through `POST /insights/regenerate`.
The in-app feature status registry now describes the full cached narrative insights view instead of the earlier Q-41-only slice.

## Why It Changed

FR-4 requires cached narrative insights to be visible in the app with citations and user-triggered regeneration.
JT-187 completes the frontend page behavior after the backend insights routes landed in JT-186.
The implementation keeps the LLM in the narrative synthesis path only and does not introduce dashboard counts, telemetry, shared credentials, auto-apply behavior, or outbound email behavior.

## Usage Notes

Open `/insights` from the primary navigation.
Cached rows appear in the matching Tier 5 cards.
Stale records show a `Stale cache` badge so the user can explicitly regenerate that insight.
Citation IDs are shown separately from the narrative text so the user can see which application, event, or email evidence grounds the cached content.
Application citations link to the existing application detail route when the citation token includes an application ID.
Regeneration is per insight type and replaces only the returned card in local page state.
Insight regeneration cost display is intentionally left to JT-188.
