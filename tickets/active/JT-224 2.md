# JT-224 - Stabilize end-to-end diagnostic baseline

## Context

The platform can sync email and several backend services exist, but the working product does not reliably prove that synced email becomes classified applications, application events, deterministic metrics, and visible frontend data.
The immediate failure reported by the user is that emails are extracted but application and rejection counts remain unavailable or zero.
This ticket creates the debugging baseline that every later repair must satisfy.

## Linked Requirements

- FR-1.5 Sync triggers.
- FR-2.1 through FR-2.7 classification, extraction, aggregation, and eval gates.
- FR-3.1 through FR-3.6 deterministic dashboard metrics.
- FR-6.1 local-first storage.
- NFR-4 reliability.
- NFR-5 maintainability.

## Scope In

- Add an end-to-end diagnostic test path that starts with representative synced email data and proves the full path into `applications`, `application_events`, and metrics.
- Add or update synthetic fixture loading so at least one application confirmation, one rejection, one interview invite, one offer, and one ghostable application are represented.
- Add a developer-facing diagnostic command or test target that reports raw email count, retained candidate count, classification count, application count, event count, rejection count, and metric summary.
- Make failure output actionable enough to identify the failed stage.
- Document the exact command sequence for local reproduction.

## Scope Out

- Do not build new product UI beyond minimal diagnostic affordances.
- Do not send real email content to an LLM in tests.
- Do not introduce telemetry or remote logging.

## Technical Notes

- Prefer deterministic fake providers and synthetic fixtures over real Gmail or real LLM calls.
- The diagnostic must verify the source-of-truth invariant: factual counts come from `applications` and `application_events`.
- Keep API routes thin if any diagnostic API is introduced.
- Do not let an LLM produce counts.

## Acceptance Criteria

- [ ] A single documented command verifies synced email -> retained candidates -> classifications -> applications -> events -> metrics.
- [ ] The diagnostic includes at least one rejected application and proves the rejection appears in deterministic metrics.
- [ ] The diagnostic output identifies which stage failed when counts are missing.
- [ ] Relevant backend tests pass with synthetic data and fake providers.
- [ ] The test data contains no private email content or secrets.
- [ ] The diagnostic does not require network access.
