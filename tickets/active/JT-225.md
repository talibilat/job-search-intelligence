# JT-225 - Repair setup, provider configuration, and credential readiness

## Context

The current setup and provider configuration surfaces are not sufficient to make classification operational.
`POST /setup` validates temporary choices but does not persist them.
`PUT /config/providers` updates the running process but is not durable.
The frontend implies that credentials and provider settings can be configured, but there is no complete operational path for storing required secret references and validating provider readiness.

## Linked Requirements

- FR-0.1 first-run setup wizard.
- FR-0.2 encrypted secrets at rest.
- FR-0.3 provider switch.
- FR-0.5 environment-variable coverage.
- FR-6.2 bring-your-own credentials.
- NFR-8 security.

## Scope In

- Implement durable non-secret provider configuration for email provider, LLM provider, classification mode, model names, endpoint settings, and sync settings.
- Add secure credential entry or documented secret-reference storage through the existing `SecretStore` boundary.
- Ensure OAuth tokens, LLM API keys, and other secrets are encrypted at rest and never returned to the frontend.
- Update the Settings or Setup UI so the user can complete configuration without manual code edits.
- Add provider health validation that explains whether sync, classification, embeddings, and chat generation are ready.
- Make the UI distinguish missing credentials, invalid model settings, provider downtime, and local Ollama unavailability.

## Scope Out

- Do not add shared credentials.
- Do not add hosted multi-user setup.
- Do not add non-Gmail providers in this ticket.

## Technical Notes

- Preserve `SecretStore` as the only boundary for token and API-key material.
- Store durable non-secret configuration in local SQLite or a local config file, but keep the app local-first.
- Avoid reading `.env` values back into the UI if they contain secrets.
- Treat `gmail.readonly` as the required v1 Gmail scope.

## Acceptance Criteria

- [ ] A fresh local instance can reach a ready-to-sync and ready-to-classify state through the UI.
- [ ] Settings persist across backend restart.
- [ ] Secrets are stored through `SecretStore` and are not printed, logged, or returned in API responses.
- [ ] Provider health clearly reports readiness for Gmail sync, classification generation, embedding generation, and chat generation where applicable.
- [ ] The frontend displays actionable setup errors instead of generic failures.
- [ ] Config loading and update behavior is covered by backend and frontend tests.
