# JT-226 - Operationalize processing from sync to applications

## Context

Sync currently stores raw emails and retained bodies, but the primary frontend does not run classification.
The endpoint that creates `email_classifications`, `applications`, and `application_events` is hidden in a legacy developer page.
The user-facing product must guide the user through sync, classification, aggregation, ghost inference, and dashboard readiness.

## Linked Requirements

- FR-1.2 full historical backfill.
- FR-1.3 incremental sync.
- FR-1.5 sync triggers.
- FR-2.2 LLM classification.
- FR-2.3 structured extraction.
- FR-2.4 application aggregation.
- FR-2.5 ghost inference.
- FR-2.6 reproducible re-runs.
- FR-3.1 foundational counts.
- NFR-4 reliability.

## Scope In

- Add a backend processing orchestration service that can run classification in batches until the retained candidate backlog is complete or until a safe user-configured limit is reached.
- Include aggregation and ghost inference in the normal processing path after accepted classifications are stored.
- Expose processing status with candidate counts, processed counts, accepted counts, malformed counts, skipped-not-job counts, application upserts, event upserts, ghost updates, manual conflicts, cost, model, and prompt version.
- Add a primary frontend action to continue processing after sync.
- Refresh pipeline status and metrics after processing completes.
- Make the UI show whether the platform is waiting for sync, waiting for classification, processing, blocked by configuration, blocked by provider failure, or ready for dashboard review.
- Preserve idempotency for reruns.

## Scope Out

- Do not make classification automatic without user-visible cost and provider context.
- Do not bypass golden-set eval requirements when classification behavior changes.
- Do not let LLM output determine dashboard counts.

## Technical Notes

- Keep orchestration in a service layer.
- Keep routes thin and return typed DTOs.
- The frontend should call stable API boundary functions, not generated files directly.
- A long-running operation may be implemented as a resumable job or controlled repeated batches, but it must expose enough status for the UI.

## Acceptance Criteria

- [ ] The primary frontend can move from synced emails to populated applications without visiting legacy developer pages.
- [ ] Processing all pending retained candidates is possible through the current UI.
- [ ] Counts in `applications` and `application_events` update after processing.
- [ ] Ghost inference runs as part of the normal processing flow or is clearly triggered by the processing flow.
- [ ] Manual conflicts are preserved and surfaced as counts for follow-up.
- [ ] Processing can be safely rerun without duplicate applications or events.
- [ ] Backend tests cover batch completion, idempotency, malformed results, and zero-candidate behavior.
- [ ] Frontend tests cover the post-sync next action and processing action states.
