# JT-227 - Correct metric truth and reconciliation

## Context

The backend has deterministic metric endpoints, but several metric semantics and reconciliation paths need correction before the dashboard can be trusted.
Application counts, submitted-application semantics, rejection counts, ghost counts, source breakdowns, company type breakdowns, and filtered response-versus-silence behavior must reconcile with the canonical tables.

## Linked Requirements

- FR-2.4 application aggregation.
- FR-2.5 ghost inference.
- FR-3.1 foundational counts.
- FR-3.2 rates, funnels, and time.
- FR-3.3 segmentation.
- FR-3.4 diagnostics.
- FR-3.5 filtering.
- FR-3.6 metric integrity.

## Scope In

- Define and enforce the difference between applications submitted, recruiter outreach, follow-up messages, and other job-related emails.
- Correct rejection counts to answer the product question using rejection evidence, not only current status, unless the PRD definition is explicitly revised.
- Ensure response-versus-silence accepts the same filters as other dashboard metrics.
- Ensure ghost inference and ghost metrics are consistent with the configured threshold.
- Populate and trust source, sponsorship, work mode, salary, seniority, tech stack, and company type only when evidence supports them.
- Fix or remove unsupported breakdown dimensions from the frontend contract.
- Add reconciliation tests proving displayed metric outputs match deterministic database facts.
- Add sample-data checks for the full dashboard metric set.

## Scope Out

- Do not use LLMs for counts, rates, funnels, or group-bys.
- Do not add external benchmarking or predictive metrics.
- Do not build chat in this ticket.

## Technical Notes

- Metric corrections must be deterministic SQL or typed Python logic.
- Any aggregation behavior change must verify idempotency and no duplicate applications.
- If classification prompt or extraction schema changes, run the relevant golden-set eval.

## Acceptance Criteria

- [ ] Q-01 through Q-21 reconcile with fixture database rows.
- [ ] Rejection count matches distinct applications with rejection evidence or a documented revised definition.
- [ ] Response-versus-silence composes with dashboard filters.
- [ ] Ghost counts update after normal processing.
- [ ] Unsupported dimensions are either implemented or removed from the UI contract.
- [ ] Deterministic tests cover totals, rejections, response/silence, funnels, trend, and filtered cases.
- [ ] Metrics do not call LLM providers.
