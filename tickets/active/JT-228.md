# JT-228 - Make the primary frontend operational

## Context

The current redesigned frontend shows a polished shell but misses operational controls and data readiness context.
Users can sync emails, but they cannot see the real pipeline stage, run classification from the primary flow, or understand why dashboard numbers are zero.
Important metrics and fields are hidden or only present in legacy pages.

## Linked Requirements

- FR-1.5 sync triggers.
- FR-2.8 manual correction path.
- FR-3.1 through FR-3.6 deterministic dashboard.
- FR-4.1 through FR-4.3 cached insights.
- NFR-5 maintainability.

## Scope In

- Add stage-aware readiness states to the current Overview and Applications flows.
- Show pipeline status, next action, and processing controls in the primary UI.
- Add prominent application, rejection, response, interview, offer, ghost, and live-application KPIs.
- Port required filters into route query strings for date range, status, source, sponsorship, role, salary band, and work mode.
- Replace static question-catalog status with actual answer availability and values where endpoints exist.
- Port complete diagnostics, response-rate trend, response-versus-silence, and complete breakdown views from legacy surfaces.
- Display application segmentation fields returned by the backend, including source, salary, location, work mode, seniority, sponsorship, and tech stack.
- Remove or repair misleading links such as `Missing one?` until manual creation exists.
- Ensure all current frontend buttons either perform real backend work or are clearly disabled as future scope.

## Scope Out

- Do not build chat UI in this ticket.
- Do not add non-Gmail providers.
- Do not add manual application creation unless explicitly approved as part of a correction workflow.

## Technical Notes

- Preserve existing visual language while making operational state explicit.
- Prefer shared accessible primitives from `frontend/src/components/ui` where available.
- Every displayed number must point to a deterministic endpoint and reconcile with the backend fixture checks.
- Avoid implying precision when pipeline stages are incomplete.

## Acceptance Criteria

- [ ] A user can tell whether the app needs Gmail connection, sync, processing, correction review, or dashboard review.
- [ ] The current frontend exposes the processing action that turns retained candidates into applications.
- [ ] Zero values are not shown as final truth when processing is incomplete.
- [ ] Rejections are visible in the primary metric hierarchy.
- [ ] Dashboard filters compose consistently and are represented in route query strings.
- [ ] Current frontend no longer depends on legacy routes for core operation.
- [ ] Frontend tests cover pipeline readiness, processing actions, metric rendering, filters, and empty states.
