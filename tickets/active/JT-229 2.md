# JT-229 - Expose corrections, conflicts, and data repair workflows

## Context

Aggregation mistakes are expected in this product.
The backend already supports several correction operations, but important controls are hidden in legacy pages or absent from the current UI.
Correction conflicts can be recorded but are not shown to the user.

## Linked Requirements

- FR-2.4 application aggregation.
- FR-2.8 manual correction path.
- FR-3.1 per-application status.
- FR-3.6 metric integrity.

## Scope In

- Port merge, split, reset-lock, status edit, and event edit workflows into the current application detail experience.
- Surface correction conflicts from `GET /applications/{id}/correction-conflicts`.
- Add a correction-history view or endpoint if needed so audited changes are reviewable.
- Make manual locks understandable and reversible.
- Preserve corrected grouping and status from automatic overwrite by default.
- When new evidence conflicts with a manual correction, show the conflict instead of silently changing the corrected record.
- Add tests for frontend correction workflows and backend conflict visibility.

## Scope Out

- Do not build autonomous email or auto-apply behavior.
- Do not introduce multi-user auditing.
- Do not let manual correction bypass source-of-truth constraints.

## Technical Notes

- Keep corrections auditable in `application_corrections`.
- Use typed request and response DTOs.
- Validate event source email IDs and date values.
- Recalculate deterministic metrics after correction changes.

## Acceptance Criteria

- [ ] Current application detail supports status edit, event edit, merge, split, and reset-lock workflows.
- [ ] Current application detail displays manual lock state and conflict records.
- [ ] Correction operations refresh affected application and metric data.
- [ ] Corrections are audited and visible to the user.
- [ ] Aggregation reruns do not silently overwrite manually locked corrections.
- [ ] Tests cover success, validation errors, conflicts, and refresh behavior.
