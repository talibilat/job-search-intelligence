# JT-231 - Ship chat UI and release hardening

## Context

After the primary frontend is operational and the chat backend exists, the disabled chat drawer must become a working product surface.
The release must verify the user can sync, process, view metrics, inspect applications, regenerate insights, and ask grounded chat questions in one local session.

## Linked Requirements

- FR-3.1 through FR-3.6 deterministic dashboard.
- FR-4.1 through FR-4.3 cached insights.
- FR-5.4 streaming chat UI.
- FR-5.5 grounded answers.
- FR-5.6 persisted chat history.
- FR-6.3 data deletion.
- Q-47 through Q-50.

## Scope In

- Replace the disabled chat drawer with a functional chat UI.
- Stream or progressively render assistant responses.
- Show citations clearly and link citations to applications or email evidence where safe.
- Display chat history from local SQLite.
- Add loading, retry, provider-unavailable, empty-history, and grounded-refusal states.
- Add a release smoke path covering setup readiness, sync, processing, dashboard counts, application detail, insights, chat, and wipe-data behavior.
- Remove or clearly mark legacy routes that are no longer needed for core operation.
- Update user-facing docs for the new operational path.

## Scope Out

- Do not add outbound email writing.
- Do not add phone or voice access.
- Do not add hosted or multi-user behavior.

## Technical Notes

- Preserve the current redesign visual language.
- Chat citations must never reveal raw secrets or unrelated private email content.
- Quantitative chat answers must match dashboard metrics.
- The UI must not imply unsupported certainty.

## Acceptance Criteria

- [ ] The chat UI can send a question and render a grounded answer.
- [ ] Responses stream or visibly progress.
- [ ] Citations are shown and navigable where applicable.
- [ ] Chat history loads after refresh.
- [ ] Quantitative chat answers match the dashboard on fixture data.
- [ ] Content chat answers cite source emails or applications.
- [ ] The release smoke path passes locally.
- [ ] Documentation explains the working end-to-end flow.
