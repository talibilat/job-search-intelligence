Title: JT-026 - Define EmailProvider interface

Mini-PRD / Context:
Add the email provider Strategy interface.
This ticket contributes to the local-first job-search intelligence app by preserving the `applications` table and event timeline as the factual source of truth.

Linked requirements:
FR-0, FR-1.4, FR-6, NFR-5, NFR-8, Phase 0.
JT-026 implements the provider-interface portion of FR-1.4 while the concrete Gmail adapter was deferred to JT-058.

Scope - in:
- Define the typed `EmailProvider` protocol for provider-specific email behavior.
- Define Pydantic DTOs for authorization, stored connections, opaque sync cursors, metadata listing, and retained body fetching.
- Define typed provider errors for auth failures, expired sync cursors, and retryable provider failures.
- Add contract tests that exercise the interface through a fake provider.

Scope - out:
- Gmail adapter skeleton implementation is covered by JT-058.
- Gmail auth-start is owned by JT-055; Gmail callback handling was deferred to JT-056 and later implemented there.
- Gmail token refresh is deferred to JT-057.
- Concrete provider message listing, metadata normalization, body fetching, sync service orchestration, and repository writes are deferred to later Phase 1 tickets.
- Provider registry work is deferred to JT-028.

Implementation notes:
- Added `backend/app/providers/` and `backend/app/providers/email/` package exports.
- Added `backend/app/providers/email/provider.py` with the `EmailProvider` runtime-checkable protocol.
- Added typed DTOs for auth start, auth callback, connection metadata, opaque provider cursors, metadata pages, message references, retained body batches, and provider capabilities.
- Added `EmailAccountRef` so provider and account identity travel through one DTO instead of repeated primitive fields.
- Kept OAuth token values out of DTOs by carrying only `SecretRef` values in `EmailConnection`.
- Kept OAuth callback authorization codes redacted at the provider seam with `SecretStr`.
- Separated metadata listing from body fetching so Phase 1 can backfill broad metadata while retaining bodies only for candidate or reconciliation messages.
- Redacted retained body text from `EmailMessageBody` repr output while preserving normal field access for the pipeline.
- Kept body-derived snippets out of broad metadata by rejecting extra metadata fields on `EmailMessageMetadata`.
- Added `EmailMetadataListRequest` validation so incremental sync requires a cursor and full backfill does not accept one.
- Enforced the v1 attachment policy as `EmailAttachmentPolicy.IGNORED` without exposing attachment content on body DTOs.
- Added typed provider errors for auth failures, expired cursors, and retryable provider failures with public-safe messages.
