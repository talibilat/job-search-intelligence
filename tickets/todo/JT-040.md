Title: JT-040 - Add insights page shell

Mini-PRD / Context:
The original empty insights route shell has been superseded by the JT-190 backend-backed Q-41 recurring-feedback insights view.
This ticket contributes to the local-first job-search intelligence app by preserving the `applications` table and event timeline as the factual source of truth.
The current `/insights` page loads cached insight data through `GET /insights` and can regenerate the Q-41 recurring-feedback insight through `POST /insights/regenerate`.

Linked requirements:
FR-0, FR-6, NFR-5, NFR-8, Phase 0

Scope - in:

- Add a minimal frontend route switch for `/insights` without introducing a router dependency.
- Keep the `/insights` route available in the frontend app.
- Document that the current page is no longer a static Phase 0 placeholder after JT-190.
- Add or update a frontend unit test proving the `/insights` page renders cached Q-41 recurring feedback.

Scope - out:

- Do not introduce telemetry, shared credentials, autonomous outbound email, auto-apply behavior, or multi-user SaaS assumptions.
- Do not add broad abstractions outside the architecture contract.
- Do not move later-phase work into this ticket unless explicitly listed.
- Do not let narrative insight generation produce authoritative dashboard counts.

Technical approach:

- Reuse the existing `App` shell and styles.
- Normalize `window.location.pathname` enough to select `/insights` while leaving full routing to later frontend work.
- Use the generated frontend API client for cached insight reads and explicit Q-41 regeneration.
- Planned files/modules: `frontend/src/App.tsx`, `frontend/src/pages/Insights.tsx`, `frontend/src/App.test.tsx`, and relevant documentation.
